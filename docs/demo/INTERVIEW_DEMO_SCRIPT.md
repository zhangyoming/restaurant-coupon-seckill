# 面试讲解稿：餐饮门店优惠券秒杀系统

## 1. 30 秒项目介绍

这个项目是我围绕餐饮门店营销场景做的优惠券秒杀系统。餐饮门店在开业或者会员日会发放限量优惠券，用户会在活动开始瞬间高并发抢券。系统主要解决高并发下的库存超卖、重复领取、数据库压力、接口防刷和异步发券问题。

技术上我使用 Spring Boot、MyBatis Plus、MySQL、Redis、Lua、RabbitMQ 和 JMeter。核心链路是：活动发布后把库存预热到 Redis，用户抢券前先通过验证码获取秒杀令牌，抢券时使用 Redis Lua 原子判断库存和重复领取并预扣库存，然后创建排队订单并投递 RabbitMQ，消费者异步扣数据库库存、创建领取记录并更新抢券结果。

---

## 2. 为什么不用数据库直接扣库存？

如果所有用户请求都直接访问 MySQL，会导致数据库在活动开始瞬间承受大量并发更新库存请求，容易出现性能瓶颈。虽然可以用 `UPDATE ... WHERE available_stock > 0` 防止超卖，但数据库压力太大。

所以我把活动库存预热到 Redis，抢券时先在 Redis 层完成库存判断和预扣，只有通过 Redis Lua 的请求才会进入后续 MQ 和数据库落库流程，大量无效请求可以在 Redis 层快速失败。

---

## 3. 为什么使用 Redis Lua？

如果用普通 Redis 命令，判断库存、判断用户是否领取、扣库存、记录用户是多个命令，并发下可能出现竞态问题。

Lua 脚本在 Redis 服务端执行，可以把这些操作合并成一个原子过程：

```text
判断库存是否大于 0
判断用户是否已经领取
扣减库存
把用户加入已领取集合
```

这样可以避免 Redis 层的超卖和重复领取。

---

## 4. Redis 扣库存成功但数据库失败怎么办？

我做了补偿机制。如果 Redis Lua 预扣库存成功，但后续创建排队订单、数据库扣库存或创建领取记录失败，系统会把 Redis 已领取用户集合里的用户移除，并把 Redis 库存加回去。

补偿时不是无条件加库存，而是先判断是否成功从已领取集合移除用户，只有移除成功才回补库存，避免重复补偿导致库存变多。

---

## 5. 为什么使用 RabbitMQ？

抢券是高并发入口，如果接口同步完成数据库扣库存和创建领取记录，请求耗时会变长，数据库压力也会集中爆发。

我使用 RabbitMQ 做异步发券：抢券接口只负责 Redis Lua 预扣库存、创建排队订单、投递消息，然后立即返回“排队中”。消费者异步处理扣数据库库存和创建领取记录。这样可以削峰填谷，保护数据库。

---

## 6. MQ 重复消费怎么处理？

消费者处理消息前会先查询秒杀订单状态。只有 `status = 0`，也就是排队中订单，才允许继续处理。

如果订单已经是成功或失败，说明之前已经处理过，当前重复消息会直接忽略。

另外，数据库里 `coupon_record` 和 `coupon_seckill_order` 都有 `user_id + activity_id` 唯一索引，作为防重复发券的最终兜底。

---

## 7. 如何防止同一用户重复领取？

一共有三层防护：

```text
第一层：Redis Lua 通过 SISMENBER 判断用户是否已经在已领取集合中。
第二层：秒杀订单表 user_id + activity_id 唯一索引防重复请求。
第三层：领取记录表 user_id + activity_id 唯一索引防重复发券。
```

这样即使 Redis 或 MQ 出现重复请求，数据库也能兜底。

---

## 8. 秒杀令牌解决什么问题？

秒杀令牌主要解决用户提前构造抢券请求和机器人直接刷接口的问题。

用户必须先获取验证码，再提交验证码换取短期有效、一次性使用的秒杀令牌。抢券接口会校验 token 是否存在、是否过期、是否绑定当前用户和活动，并且会一次性删除 token。

这样即使别人知道抢券接口地址，也不能在没有 token 的情况下直接刷接口。

---

## 9. 限流怎么做？

我做了用户维度和 IP 维度的固定窗口限流，使用 Redis Lua 保证 `INCR + EXPIRE` 原子执行。

包括：

```text
抢券接口用户限流
抢券接口 IP 限流
创建秒杀令牌限流
获取验证码限流
```

固定窗口实现简单，适合当前项目。后续可以升级成滑动窗口限流。

---

## 10. 如何验证没有超卖？

压测后执行 SQL：

```sql
SELECT COUNT(*) FROM coupon_record WHERE activity_id = 9001;
SELECT COUNT(*) FROM coupon_seckill_order WHERE activity_id = 9001 AND status = 1;
SELECT total_stock, available_stock FROM coupon_activity WHERE id = 9001;
```

需要满足：

```text
领取记录数 = 成功订单数
领取记录数 + 剩余库存 = 总库存
```

如果满足，说明没有超卖，也没有重复发券。

---

## 11. 项目亮点总结

这个项目的亮点不是普通优惠券 CRUD，而是完整的高并发抢券链路：

```text
Redis 库存预热减少数据库访问
Redis Lua 原子扣库存防超卖和重复领取
RabbitMQ 异步发券削峰填谷
订单状态幂等和唯一索引防重复消费
验证码、秒杀令牌、用户/IP 限流防刷
JMeter 压测和 SQL 数据一致性校验
```
